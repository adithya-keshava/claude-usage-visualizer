# Data Loading & Aggregation Flows

## Overview

The data loading pipeline fetches raw Claude session files and aggregates them into progressively coarser statistics for dashboards. Three levels of loading are supported:

1. **Raw file loading** - Parse JSON/JSONL files from disk
2. **Session aggregation** - Combine messages into session summaries
3. **Project aggregation** - Combine sessions into project summaries
4. **Dashboard aggregation** - Load pre-computed overview statistics

**File Reference:** `src/app/data/loader.py`

## Data Source Files

### history.jsonl

**Location:** `~/.claude/history.jsonl`

Session metadata and display names (one JSON object per line).

**Format:** JSONL with structure:

```json
{"sessionId": "abc123", "project": "/IdeaProjects/my-project", "timestamp": "2026-02-20T14:30:00Z", "display": "How do I debug async code?"}
{"sessionId": "abc123", "project": "/IdeaProjects/my-project", "timestamp": "2026-02-20T14:30:00Z", "display": "/rename User-Friendly Name"}
```

**Fields:**
- `sessionId`: Unique session identifier (used as filename key)
- `project`: Original directory path where session was created
- `timestamp`: Session creation timestamp
- `display`: Either session first message (user-friendly name) or `/rename {new_name}` command

**Rename command handling** at `src/app/data/loader.py:135-158`:

If `display` starts with `/rename`:
- Extract renamed text: `display[7:].strip()`
- Update history entry with `"renamed": True`
- Use renamed text as session slug in UI

**Purpose:** Provide human-friendly session names and project context

**Loading function:** `load_history()` at `src/app/data/loader.py:104-167`

### Session JSONL Files

**Location:** `~/.claude/projects/{encoded_path}/{session_id}.jsonl`

Individual assistant messages with token counts (one JSON object per line).

**Format:** JSONL with message objects:

```json
{"type": "assistant", "timestamp": "2026-02-20T14:30:05Z", "message": {"model": "claude-opus-4-6", "usage": {"input_tokens": 100, "output_tokens": 250, "cache_creation_input_tokens": 0, "cache_read_input_tokens": 0}}, "slug": "session-slug"}
{"type": "user", "timestamp": "2026-02-20T14:30:06Z", "content": "What's the answer?"}
{"type": "assistant", "timestamp": "2026-02-20T14:30:10Z", "message": {"model": "claude-opus-4-6", "usage": {...}}, "isSidechain": false}
```

**Relevant fields for token counting:**
- `type`: Only process "assistant" messages
- `message.model`: Claude model ID (e.g., "claude-opus-4-6")
- `message.usage`: Token object with input_tokens, output_tokens, cache_creation_input_tokens, cache_read_input_tokens
- `isSidechain`: Boolean flag for abandoned branches (skip if true)

**Field name handling** at `src/app/data/loader.py:170-177`:

Handles both camelCase (API) and snake_case (file storage) field names:
```python
def _parse_message_usage(usage_dict: dict) -> dict:
    return {
        "input_tokens": usage_dict.get("input_tokens", 0),
        "output_tokens": usage_dict.get("output_tokens", 0),
        "cache_creation_input_tokens": usage_dict.get("cache_creation_input_tokens", 0),
        "cache_read_input_tokens": usage_dict.get("cache_read_input_tokens", 0),
    }
```

**Purpose:** Detailed message-by-message breakdown for session detail view

**Loading function:** `load_session_messages()` at `src/app/data/loader.py:180-253`

### Subagent Files

**Location:** `~/.claude/projects/{encoded_path}/{session_id}/subagents/agent-*.jsonl`

Assistant messages from spawned subagents (merged into parent session).

**Format:** Same as session JSONL files

**Merging behavior** at `src/app/data/loader.py:317-345`:

When building session summary:
1. Check if `projects/{encoded_path}/{session_id}/subagents/` directory exists
2. For each `agent-*.jsonl` file:
   - Load messages same way as main session
   - Parse token usage and calculate cost
   - Add tokens to session totals
   - Add models to session's models_used set

**Purpose:** Account for work done by spawned agents in same session context

## Loading Flows

### Flow 1: Load Stats Cache (Dashboard Overview)

**Entry point:** `src/app/routers/overview.py:18-96`

```
User visits GET /
  ↓
overview() router handler
  ↓
load_stats_cache()  [src/app/data/loader.py:21-101]
  ├─ Check cache (key: "stats_cache")
  │  └─ If valid, return cached OverviewStats
  │
  └─ Load file: stats-cache.json
     ├─ Parse JSON
     ├─ Build DailyActivity list
     ├─ Build DailyModelTokens list
     ├─ Build ModelStats dict
     ├─ Create OverviewStats object
     ├─ Cache result (5-min TTL + mtime tracking)
     └─ Return OverviewStats or None

  ↓
Process OverviewStats in router:
  ├─ Calculate total cost: sum(estimate_cost() for each model)
  ├─ Format model_rows for display
  └─ Render template with data

  ↓
Return HTML with overview dashboard
```

**Caching:** Key "stats_cache" with mtime-based invalidation

**Performance:** Direct file → object mapping, no recursive aggregation

**Failure mode:** Return None, render error message in template

### Flow 2: Load Project Summaries (Projects List & Detail)

**Entry point:** `src/app/routers/projects.py:17-86` (list) or `src/app/routers/projects.py:89-155` (detail)

```
User visits GET /projects
  ↓
projects_list() router handler
  ↓
build_project_summaries()  [src/app/data/loader.py:360-395]
  ├─ Check cache (key: "project_summaries")
  │  └─ If valid, return cached {encoded_path: [SessionSummary, ...]}
  │
  └─ Scan projects directory:
     └─ For each subdirectory (encoded_path):
        └─ For each *.jsonl file (session_id):
           └─ build_session_summary(encoded_path, session_id, history)
              ├─ load_session_messages(encoded_path, session_id)
              │  ├─ Load session JSONL file
              │  ├─ Parse assistant messages
              │  ├─ Skip synthetic/sidechain
              │  ├─ Extract model, tokens, calc cost
              │  └─ Return [SessionMessage, ...]
              │
              ├─ Aggregate message data:
              │  ├─ Sum input_tokens
              │  ├─ Sum output_tokens
              │  ├─ Sum costs
              │  └─ Collect models_used
              │
              ├─ Determine slug (rename > display > file > prefix)
              │
              ├─ Load subagents (if exist):
              │  └─ For agent-*.jsonl files:
              │     ├─ Parse messages
              │     ├─ Add tokens to totals
              │     └─ Add models to set
              │
              └─ Return SessionSummary

        └─ Collect all SessionSummary objects
           └─ Store in dict: {encoded_path: [SessionSummary, ...]}

  ├─ Cache result
  └─ Return {encoded_path: [SessionSummary, ...]}

  ↓
Process in router:
  ├─ Aggregate each project:
  │  ├─ Sum session costs
  │  ├─ Count sessions
  │  └─ Calculate date range
  │
  └─ Format for display

  ↓
Return HTML with project list or detail page
```

**Caching:** Key "project_summaries" with 5-min TTL

**Sub-caching:** Each session is cached separately (key: "session_{encoded_path}_{session_id}")

**Performance:** O(P × S) where P = projects, S = sessions per project

**Failure mode:** Return empty dict, render empty list or error message

### Flow 3: Load Session Messages (Session Detail)

**Entry point:** `src/app/routers/sessions.py:16-93`

```
User visits GET /sessions/{encoded_path}/{session_id}
  ↓
session_detail() router handler
  ├─ build_session_summary() [uses flow from Flow 2]
  │  └─ Returns SessionSummary for metadata
  │
  └─ load_session_messages()  [src/app/data/loader.py:180-253]
     ├─ Check cache (key: "session_{encoded_path}_{session_id}")
     │  └─ If valid, return cached [SessionMessage, ...]
     │
     └─ Load session JSONL file:
        ├─ For each line in file:
        │  ├─ Parse JSON
        │  ├─ Filter:
        │  │  ├─ Only type == "assistant"
        │  │  ├─ Skip model == "<synthetic>"
        │  │  └─ Skip isSidechain == true
        │  │
        │  └─ Extract and create SessionMessage:
        │     ├─ timestamp
        │     ├─ model
        │     ├─ input_tokens
        │     ├─ output_tokens
        │     ├─ cache_creation_input_tokens
        │     ├─ cache_read_input_tokens
        │     └─ cost_usd = estimate_cost(...)
        │
        ├─ Cache result
        └─ Return [SessionMessage, ...]

  ↓
Format messages for display:
  ├─ For each message:
  │  ├─ Extract timestamp
  │  ├─ Format date_time
  │  ├─ Include model, tokens, cost
  │  └─ Add to message_rows
  │
  └─ Calculate totals (sum tokens, costs, etc.)

  ↓
Return HTML with message table
```

**Caching:** Key "session_{encoded_path}_{session_id}" with 5-min TTL

**Performance:** O(M) where M = messages in session (usually 10-100)

**Failure mode:** Return empty list, render empty message table

### Flow 4: Hourly Activity (Smart Granularity)

**Entry point:** `src/app/routers/api.py:312-407`

**Conditional granularity:**

```
User requests /api/activity?start_date=X&end_date=Y
  ↓
get_activity() [src/app/routers/api.py:312-407]
  ├─ Parse dates and calculate duration
  │
  └─ If duration < 1 day:
     │
     └─ Hourly granularity:
        └─ build_hourly_activity(start_date, end_date, project)
           [src/app/data/loader.py:398-445]
           ├─ Load project_summaries()
           │
           └─ For each project (or filtered project):
              └─ For each session in date range:
                 └─ load_session_messages() for detailed timestamps
                    └─ For each message:
                       ├─ Extract hour from timestamp: "2026-02-20T14:30:00Z" → "2026-02-20T14"
                       ├─ Add to hourly_data[hour_key]
                       │  ├─ messages += 1
                       │  ├─ sessions.add(session_id)
                       │  └─ cost += message.cost_usd
                       │
                       └─ Return: [{"hour": key, "message_count": N, "session_count": N, "total_cost": X}, ...]

  └─ Else: >= 1 day:
     │
     └─ Daily granularity:
        └─ load_stats_cache()
           └─ Filter daily_activity by date range
              └─ Return [DailyActivity, ...]

  ↓
Format for Chart.js:
  ├─ labels: List of hours (or dates)
  ├─ datasets: Messages and sessions counts
  └─ Return JSON
```

**Performance:**
- Hourly: O(P × S × M) but filtered to 1 day
- Daily: O(D) where D = days (from cache)

**Use case:** Sub-day analysis needs message-level detail, multi-day uses pre-computed cache

## Caching Strategy

### Cache Implementation

**Type:** In-memory Python dict in global variable `_cache` (see `src/app/data/cache.py`)

**TTL:** 5 minutes for all entries (see `src/app/data/cache.py:5`)

**Invalidation:** File mtime tracking (see `src/app/data/cache.py:8-26`)

```python
class CacheEntry:
    def is_valid(self) -> bool:
        if time.time() - self.cached_at > CACHE_TTL:
            return False
        if self.file_path and self.file_path.exists():
            if self.file_path.stat().st_mtime != self.cached_mtime:
                return False
        return True
```

### Cache Keys

| Key | Data Type | Size Estimate | TTL | Invalidated By |
|-----|-----------|---------------|-----|----------------|
| "stats_cache" | OverviewStats | ~1KB | 5m | stats-cache.json mtime |
| "history" | dict[str, dict] | ~50KB | 5m | history.jsonl mtime |
| "project_summaries" | dict[str, list[SessionSummary]] | ~500KB | 5m | Any session file mtime |
| "session_{encoded_path}_{session_id}" | list[SessionMessage] | ~10-100KB | 5m | session JSONL mtime |

### Cache Hit Scenarios

**Best case:** Same user requests same page within 5 minutes
- Example: User views overview dashboard, refreshes → cache hit

**Common case:** Multiple API calls for same data within 5 minutes
- Example: Frontend makes 4 separate chart requests → 1st loads, 3-4 hit cache

**Worst case:** User changes data directory or files are updated externally
- Cache invalidation waits for TTL (5 minutes) or mtime change detection
- Solution: Reload page or wait for TTL expiry

## Error Handling in Loading

### File Not Found

**Example:** `src/app/data/loader.py:35-37`

```python
stats_path = get_data_dir() / "stats-cache.json"
if not stats_path.exists():
    logger.debug("stats-cache.json not found")
    return None
```

**Behavior:** Return None, caller handles gracefully

**Router response:** Render error message in template

### Malformed JSON

**Example:** `src/app/data/loader.py:159-161`

```python
except json.JSONDecodeError as e:
    logger.warning(f"history.jsonl line {line_num}: malformed JSON: {e}")
    continue
```

**Behavior:** Log warning, skip line, continue processing

**Result:** Partial data loaded (skip bad entries)

### Parse Error (Schema Mismatch)

**Example:** `src/app/data/loader.py:99-101`

```python
except (KeyError, ValueError, TypeError) as e:
    logger.error(f"Failed to parse stats-cache.json: {e}")
    return None
```

**Behavior:** Catch schema mismatches, log error, return None

**Router response:** Render error message

## Performance Characteristics

### Load Times

| Operation | Time (Cached) | Time (Cache Miss) |
|-----------|:-------------:|:----------------:|
| Overview dashboard | ~10ms | 50-200ms |
| Projects list | ~50ms | 500ms - 2s |
| Project detail | ~20ms | 300-500ms |
| Session detail | ~5ms | 100-300ms |

Cache hit based on 5-minute TTL.

### Memory Usage

- stats_cache: ~1 KB
- history: ~50 KB (1000 sessions)
- project_summaries: ~500 KB (100 projects, 1000 sessions total)
- session messages: ~10-100 KB each (cached up to 5 min × concurrent users)

**Total:** ~500 KB baseline + 10 KB per session in flight

## Future Optimizations

1. **Database backend:** Replace file-based loading with SQLite/PostgreSQL
2. **Incremental updates:** Only rescan changed projects instead of full rebuild
3. **Lazy subagent loading:** Load subagent files on-demand instead of per-session
4. **Compression:** Store cached objects in compressed format
5. **Distributed cache:** Use Redis for multi-instance deployments
